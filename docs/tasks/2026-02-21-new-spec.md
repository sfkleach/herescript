# New conventions!

After some thought we have stumbled over a better design. We will initially
implement in parallel with the existing `#!` prefix but once established we
will retire the old approach.

The core ideas are:

 1. that we will use "shell-like" parsing of the header lines.
 2. that the header lines will start with `##`, `#:`, or `#|`.
    - `##` for a comment line
    - `#:` for an arguments line
    - `#|` for a binding
    - `#>` for an in-line quoted argument

These can be freely interleaved with the existing `#!` header lines. It is a 
separate task to remove those. The header blocks ends with a non-header line.

## Step 1, Implement the new comment line syntax

Recognise lines starting `##` as header lines that are discarded.

## Step 2, Implement the new arguments line

Recognise lines starting `#:` as a new type of header line. The following 
basic, common shell-like tokenisation rule should be followed.

### Step 2a, whitespace

- Runs of whitespace separate tokens. Ignore quoting and variables for now.
- Each token becomes an argument.

### Step 2b, single quotes

- Single quotes enclose literal text. Backslash escapes are not enabled, nor interpolations.

### Step 2c, single quote escapes enabled

- $'...' enable full backslash quoting within the string.
- Backslash may be followed by 
  - another backslash
  - a single or double quote
  - the usual whitespace escapes \s, \n, \t, \r

### Step 2d, ${...} variables

- Variables are written `${NAME}`, defaults to come later.
- Outside of quotes, the environment variable is substituted.

The braces around the arguments will be made optional (where practical) in a
later step.

### Step 2e, ${<number>} parameters

- Parameters are written `${1}`, `${2}` for specific parameters. Parameters are
  the additional arguments supplied by the user when they run the shebang script
  which appear after the script-file name.
- The special parameter `${0}` is the script-name itself, expanded using realpath.

### Step 2f, parameter slices

- `${1:}` for all additional parameters, which is a parameter slice notation.
  - This slice notation is a replacement for `$@` and substitutes them as separate arguments.
  - It can be written `${A:B}` where A & B are integers, A <= B
  - If A or B are omitted they default to 0 and argc respectively.
  - This expands into B-A arguments.
  - Missing arguments are transformed into empty strings.

### Step 2g, ${HERESCRIPT_FILE} and ${}

The file-path is bound to the environment variable `${HERESCRIPT_FILE}`.

The expression `${}` is expanded into `${HERESCRIPT_FILE}` followed by
the additional arguments `${1:}`.

The default behaviour of appending the name of the file and the additional
arguments should be disabled at this step! All the tests will need revising.


### Step 2h, double quotes

- Double quotes support variable interpolation and minimal backslash escapes enabled for:
  - backslash
  - double quote
  - dollar
- To enable full backslash escape sequence expansion, put a $ at the front of either single or double quotes.


## Step 3, Implement the new unconditional bindings line

Recognise lines starting `#|` as bindings lines. These must be of same form
as before, namely:

```
#| NAME=TOKEN
```

There may be optional whitespace around the `=` sign and trailing whitespace.
This whitespace is discarded. To add explicit whitespace the value should be 
quoted. The value is processed using the tokenisation rules. Multiple tokens
cause a syntax error.

## Step 4, Implement the conditional bindings header line

Conditional bindings only apply when NAME is unset, analogous to bash shells
`${NAME:-VALUE}` but allowing the default to be applied in one place when the
variable is used multiple times.

```
#| NAME:-TOKEN
```

N.B. `#| NAME:-VALUE` and `${NAME:-VALUE}` (see Step 5 below) are semantically
equivalent but serve different ergonomic purposes. Use `#|` when you want a
single reviewable manifest of all variables and their defaults. Use
`${NAME:-VALUE}` inline when the default is naturally expressed at the point
of use and the variable appears only once, matching the general expectation
of bash-like behaviour in a common case.

### Step 5, Implement quoted, in-line arguments

In-line arguments are any run of header lines that start `#>` followed by
a single whitespace character. The first three characters of each line (`#> `)
are discarded and the run of lines are glued together, preserving line 
breaks. 

Note that `#:` can be used to force a break between successive arguments. 

Each of the arguments are bound to environments variables `${HERESCRIPT0}`,
`${HERESCRIPT1}`, .. etc.

### Step 6, Implement in-line defaults

Variables of the form `${NAME:-VALUE}` provide defaults at the point of use,
in-line.


## Step 7, Make braces optional in some cases

- `$NAME` is allowed for `${NAME}` as long as it is terminated by whitespace or
  end of line. Any other termination is a syntax error.
- `$<number>` is allowed for `${<number>}` as long as it is terminated by
  whitespace or end of line. Any other termination is a syntax error.


## Final Example

```
#!/usr/bin/herescript python3
## Run with verbose output.
#| PYTHONPATH=/usr/local/lib
#| LOG_LEVEL:-"info"
#: --verbose --output=/tmp/out $0
```
